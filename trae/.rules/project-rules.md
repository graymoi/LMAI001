# 项目管理规则

## 规则概述

本文档定义了项目管理系统的各项规则，包括文件管理规则、文档编写规则、代码编写规则、Git管理规则等，确保项目管理的规范性和一致性。

---

## 文件管理规则

### 1. 文件夹结构规则

#### 1.1 文件夹命名规则

**规则描述：** 文件夹命名必须清晰、规范、易于理解

**命名规范：**
- 使用数字前缀排序（如：01_、02_、03_）
- 使用下划线分隔单词
- 使用中文或英文命名
- 避免使用特殊字符

**示例：**
```
✅ 正确示例：
- 01_项目文档
- 02_工具和脚本
- 03_文档和报告

❌ 错误示例：
- 项目文档（缺少数字前缀）
- 02工具和脚本（缺少下划线）
- 03-文档和报告（使用连字符）
```

#### 1.2 文件夹组织规则

**规则描述：** 文件夹必须按照功能、项目、时间等维度组织

**组织原则：**
- 按功能分类：文档、代码、数据、工具
- 按项目分类：各项目独立文件夹
- 按时间分类：归档文件按时间组织
- 按状态分类：进行中、已完成、归档

**标准结构：**
```
f:\AIlm\000项目管理\
├── 00_系统配置\
├── 01_项目文档\
├── 02_工具和脚本\
├── 03_文档和报告\
├── 04_数据文件\
├── 05_代码文件\
├── 06_测试文件\
├── 07_部署文件\
├── 08_归档文件\
└── 99_临时文件\
```

### 2. 文件命名规则

#### 2.1 文档文件命名规则

**规则描述：** 文档文件命名必须包含日期、项目名称、文档类型、版本信息

**命名格式：** `[日期]-[项目名称]-[文档类型]-[版本].md`

**示例：**
```
✅ 正确示例：
- 2026-03-13-天津背街小巷-需求规格说明书-V1.0.md
- 2026-03-13-项目管理-总结报告-V1.0.md

❌ 错误示例：
- 需求规格说明书.md（缺少日期、项目名称、版本）
- 20260313天津背街小巷需求规格说明书V1.0.md（缺少连字符）
```

#### 2.2 代码文件命名规则

**规则描述：** 代码文件命名必须清晰描述文件功能

**命名格式：** `[模块名称]-[功能描述].py`

**示例：**
```
✅ 正确示例：
- data-analysis-visualization.py
- document-generation-template.py
- folder-automation-organize.py

❌ 错误示例：
- data.py（功能描述不清晰）
- document.py（功能描述不清晰）
- folder.py（功能描述不清晰）
```

#### 2.3 配置文件命名规则

**规则描述：** 配置文件命名必须清晰描述配置类型和名称

**命名格式：** `[配置类型]-[配置名称].json`

**示例：**
```
✅ 正确示例：
- agent-config-default.json
- hook-config-pre-commit.json
- skill-config-bidding-automation.json

❌ 错误示例：
- config.json（配置类型不清晰）
- default.json（配置名称不清晰）
- bidding.json（配置类型不清晰）
```

### 3. 文件内容规则

#### 3.1 Markdown文档规则

**规则描述：** Markdown文档必须遵循标准格式和规范

**格式要求：**
- 使用UTF-8编码
- 使用标准Markdown语法
- 使用统一的标题层级
- 使用规范的表格格式

**内容要求：**
- 文档必须有标题
- 文档必须有概述
- 文档必须有目录
- 文档必须有结论

**示例：**
```markdown
# 文档标题

## 概述
本文档描述...

## 目录
- 章节1
- 章节2
- 章节3

## 章节1
内容...

## 结论
总结...
```

#### 3.2 Python代码规则

**规则描述：** Python代码必须遵循PEP 8规范

**格式要求：**
- 使用4空格缩进
- 行长度不超过120字符
- 使用有意义的变量名
- 添加必要的注释

**示例：**
```python
# ✅ 正确示例
def calculate_total_price(items):
    """
    计算总价
    
    Args:
        items: 商品列表
        
    Returns:
        总价
    """
    total = 0
    for item in items:
        total += item['price'] * item['quantity']
    return total

# ❌ 错误示例
def calc(items):
    t=0
    for i in items:
        t+=i['p']*i['q']
    return t
```

#### 3.3 JSON配置规则

**规则描述：** JSON配置文件必须格式规范、易于阅读

**格式要求：**
- 使用2空格缩进
- 使用双引号
- 键名使用小写和下划线
- 添加必要的注释（使用comment字段）

**示例：**
```json
{
  "config_name": "示例配置",
  "version": "1.0.0",
  "enabled": true,
  "comment": "这是一个示例配置文件",
  "settings": {
    "timeout": 30,
    "retry_count": 3,
    "comment": "超时和重试设置"
  }
}
```

## 代码编写规则

### 1. 代码规范规则

#### 1.1 Python代码规范

**规则描述：** Python代码必须遵循PEP 8规范

**规范要求：**
- 使用4空格缩进
- 行长度不超过120字符
- 使用有意义的变量名
- 添加必要的注释
- 使用类型提示

**示例：**
```python
from typing import List, Dict

def process_data(data: List[Dict]) -> Dict:
    """
    处理数据
    
    Args:
        data: 数据列表
        
    Returns:
        处理后的数据字典
    """
    result = {}
    for item in data:
        key = item.get('key')
        value = item.get('value')
        if key and value:
            result[key] = value
    return result
```

#### 1.2 JavaScript代码规范

**规则描述：** JavaScript代码必须遵循ESLint规范

**规范要求：**
- 使用2空格缩进
- 使用单引号
- 使用分号
- 使用const和let

**示例：**
```javascript
// ✅ 正确示例
const processData = (data) => {
  const result = {};
  data.forEach(item => {
    const key = item.key;
    const value = item.value;
    if (key && value) {
      result[key] = value;
    }
  });
  return result;
};

// ❌ 错误示例
function processData(data) {
  var result = {};
  for (var i = 0; i < data.length; i++) {
    var item = data[i];
    result[item.key] = item.value;
  }
  return result;
}
```

### 2. 代码质量规则

#### 2.1 代码质量标准

**规则描述：** 代码必须高质量、可维护、可扩展

**质量标准：**
- 可读性：代码必须易于阅读
- 可维护性：代码必须易于维护
- 可扩展性：代码必须易于扩展
- 可测试性：代码必须易于测试

**检查方法：**
- 代码审查
- 静态分析
- 单元测试
- 集成测试

#### 2.2 代码安全规则

**规则描述：** 代码必须安全可靠，无安全隐患

**安全要求：**
- 输入验证：必须验证所有输入
- 输出编码：必须编码所有输出
- 错误处理：必须处理所有错误
- 日志记录：必须记录关键操作

**示例：**
```python
# ✅ 正确示例
import logging

def process_user_input(user_input: str) -> str:
    """
    处理用户输入
    
    Args:
        user_input: 用户输入
        
    Returns:
        处理后的输出
    """
    try:
        # 验证输入
        if not user_input or not isinstance(user_input, str):
            logging.warning("Invalid user input")
            return ""
        
        # 处理输入
        result = user_input.strip()
        
        # 记录日志
        logging.info(f"Processed user input: {result[:50]}...")
        
        return result
    except Exception as e:
        logging.error(f"Error processing user input: {str(e)}")
        return ""

# ❌ 错误示例
def process_user_input(user_input):
    return user_input
```

## Git管理规则

### 1. 提交规则

#### 1.1 提交信息规则

**规则描述：** 提交信息必须清晰、规范、有意义

**提交格式：** `<type>(<scope>): <subject>`

**类型说明：**
- feat: 新功能
- fix: 修复bug
- docs: 文档更新
- style: 代码格式调整
- refactor: 重构代码
- test: 测试相关
- chore: 构建/工具相关

**示例：**
```
✅ 正确示例：
- feat(文档): 添加需求规格说明书章节
- fix(代码): 修复文件整理bug
- docs(文档): 更新README文档

❌ 错误示例：
- 更新文档（缺少类型和范围）
- fix bug（缺少范围和详细描述）
- 添加文件（缺少类型）
```

#### 1.2 提交频率规则

**规则描述：** 提交必须频繁、小步、有意义

**提交原则：**
- 频繁提交：每天至少提交一次
- 小步提交：每次提交只做一件事
- 有意义提交：每次提交都有明确目的
- 可回滚提交：每次提交都可以独立回滚

### 2. 分支规则

#### 2.1 分支命名规则

**规则描述：** 分支命名必须清晰、规范、易于理解

**命名格式：** `<type>/<description>`

**类型说明：**
- feature: 新功能分支
- bugfix: 修复bug分支
- hotfix: 紧急修复分支
- release: 发布分支

**示例：**
```
✅ 正确示例：
- feature/需求规格说明书编写
- bugfix/文件整理问题
- hotfix/紧急文档修复

❌ 错误示例：
- feature（缺少描述）
- 需求规格说明书（缺少类型）
- fix bug（格式不规范）
```

#### 2.2 分支管理规则

**规则描述：** 分支必须规范管理、及时合并、及时清理

**管理原则：**
- 主分支保护：main分支必须保护
- 分支生命周期：分支完成后及时合并和删除
- 代码审查：合并前必须进行代码审查
- 冲突解决：及时解决合并冲突

### 3. 标签规则

#### 3.1 标签命名规则

**规则描述：** 标签命名必须遵循语义化版本规范

**命名格式：** `v<major>.<minor>.<patch>`

**版本说明：**
- major: 重大变更，不兼容旧版本
- minor: 功能新增，兼容旧版本
- patch: Bug修复，小改动

**示例：**
```
✅ 正确示例：
- v1.0.0（初始版本）
- v1.1.0（功能新增）
- v1.1.1（Bug修复）

❌ 错误示例：
- v1.0（缺少补丁版本）
- 1.0.0（缺少v前缀）
- v1.0.0.0（版本号过长）
```

## 经济价值规则

### 1. 价值创造规则

#### 1.1 效率价值规则

**规则描述：** 通过自动化提升工作效率，创造经济价值

**价值计算：**
- 时间节省 = (优化前时间 - 优化后时间) × 人力成本
- 效率提升 = (优化前效率 - 优化后效率) / 优化前效率 × 100%

**价值目标：**
- 文档生成效率提升 ≥ 50%
- 代码生成效率提升 ≥ 40%
- 文件整理效率提升 ≥ 60%

#### 1.2 质量价值规则

**规则描述：** 通过自动化提升工作质量，创造经济价值

**价值计算：**
- 质量提升 = (优化后质量 - 优化前质量) × 质量成本
- 返工减少 = (优化前返工率 - 优化后返工率) × 返工成本

**价值目标：**
- 文档质量提升 ≥ 50%
- 代码质量提升 ≥ 40%
- 返工率降低 ≥ 60%

### 2. 价值评估规则

#### 2.1 评估周期规则

**规则描述：** 定期评估自动化创造的经济价值

**评估周期：**
- 每日评估：评估每日工作效率
- 每周评估：评估每周工作质量
- 每月评估：评估每月经济价值
- 每季度评估：评估季度价值趋势

#### 2.2 评估报告规则

**规则描述：** 评估结果必须生成报告并记录

**报告内容：**
- 价值创造总结
- 效率提升数据
- 质量提升数据
- 经济价值计算
